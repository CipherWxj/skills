# PRD 质检整体评估原则

本文件只描述 `prd-quality-checker` 的评估立场，不规定执行步骤、输出字段、评分权重或具体规则。执行流程以 `SKILL.md` 为准，阶段任务以 `references/prompts/*.md` 为准，规则与扣分以 `assets/*.yaml` 为准。

## 1. 评估目标

PRD 质检不是重写需求，也不是做模板填空检查。它要回答一个问题：这份 PRD 是否足以支撑需求评审、研发理解和测试设计。

评估时同时关注三类结果：

- 是否能读懂：背景、目标、范围、方案和影响对象是否清楚。
- 是否能实现：规则、流程、字段、依赖和边界是否足够明确。
- 是否能测试：权限、输入、触发、预期、异常、指标和验收口径是否可验证。

## 2. 判断立场

先理解业务意图，再使用规则。规则用于稳定发现风险，不用于机械匹配关键词。

判断时遵守以下立场：

- 以当前 PRD 自身可复核的信息为主，外部知识只能辅助解释，不能替代当前文档缺失的信息。
- 以本期交付为边界，区分核心缺口、协作占位和后续规划。
- 以对评审、研发、测试的实际影响判定严重度，而不是以文字多少或模板完整度判定。
- 对同一根因只输出一次，避免结构问题和内容问题重复表达。

## 3. 结构与内容关系

结构评估看“骨架是否成立”：PRD 是否具备背景、范围、方案、影响、风险、验收等基本支撑。

内容评估看“骨架内的信息是否可用”：已有描述是否清晰、闭环、无关键歧义，并能拆成测试条件和预期结果。

结构不足通常意味着信息入口缺失；内容不足通常意味着信息存在但不能落地。若两者指向同一根因，最终报告应合并成一个更准确的问题。

## 4. 模板与主体的区分

不要把模板残留直接等同于 PRD 缺陷。判断一个空表、空章节或占位符是否需要扣分，取决于它是否承载本期核心判断。

如果主体内容已经能支撑理解和测试，模板占位只作为非阻断说明；如果占位处是唯一的验收、字段、埋点、实验或联调信息来源，则应作为正式问题。

## 5. 证据原则

所有正式问题都必须能回到 PRD 原文。没有原文证据时，只能作为风险提示或非阻断说明，不能作为扣分项。

证据使用要克制：引用最能说明问题的 1-2 句原文，并说明它为什么影响评审、实现或测试。最终报告应输出可复核结论，不暴露内部分析过程。
