# 步骤3: 综合评分与报告生成

## 目标

将结构问题和内容问题去重汇总，计算最终得分，输出面向用户可读的 PRD 质量检测报告。

## 输入

- `structure-review.json`
- `content-review.json`
- `assets/03-review-template.md`

## 分工边界

本 prompt 只负责综合评分、跨阶段去重、最终报告组织和 `final-score.json` 输出；不重新定义结构或内容规则。

- 结构判断标准来自 `assets/01-doc-structure.yaml` 与 `structure-review.json`。
- 内容判断标准来自 `assets/02-content-quality.yaml` 与 `content-review.json`。
- 报告章节和表格骨架来自 `assets/03-review-template.md`。
- 本阶段只对 findings 做二次治理：去重、过滤、排序、扣分、归纳待确认点和可测性结论。

## 执行步骤

### 1. 载入并规范化输入

读取两个阶段产物后，先检查字段是否完整：

- 结构 finding 至少包含：`rule_id`、`rule_name`、`status`、`severity`、`description`、`evidence`、`reasoning`
- 内容 finding 至少包含：`rule_id`、`rule_name`、`severity`、`description`、`evidence`、`reasoning`
- `non_blocking_notes` 为空时按空数组处理

将每条 finding 规范化为统一内部结构：

- `dimension`：`structure` 或 `content`
- `rule_id`
- `rule_name`
- `severity`
- `problem`：来自 `description`，必要时改写为“具体对象 + 具体缺口”
- `impact`：从 `reasoning` 中提炼，必须说明影响评审、研发、测试或上线判断的具体动作
- `confirmation_question`：把缺口转成 PRD 作者需要回答的问题
- `evidence`

### 2. 过滤非阻断项

合并两个阶段的 `non_blocking_notes`，并对 findings 做二次过滤。符合“模板占位过滤规则”的 finding：

- 不进入问题表
- 不参与扣分
- 不计入命中统计
- 计入 `filtered_count`
- 可合并写入报告的“非阻断说明”

### 3. 去重与合并

对剩余 findings 按根因去重，而不是按规则编号机械去重。

合并优先级：

1. 同一 PRD 原文证据、同一缺口、同一待确认问题，合并为一条。
2. 结构问题和内容问题描述同一根因时，保留更能解释测试风险的一条。
3. 合并后的严重度取最高严重度；扣分只扣一次。
4. 保留所有关联 `rule_id` 用于追溯，但报告表格只展示一个最符合主风险的规则编号，禁止用逗号连接多个规则编号。
5. 选择展示规则编号时，先判断合并后问题的主风险：
   - 主风险是结构准入缺口，例如缺少背景、范围、完成口径入口、实验结构、风险依赖入口，展示对应 `structure-*` 规则。
   - 主风险是可实现性或可测性缺口，例如字段口径、流程闭环、异常分支、上下游契约、影响面细节、指标口径，展示对应 `content-*` 规则。
   - 当结构规则和内容规则都能解释同一根因时，选择更贴近“问题 / 影响 / 待确认点”的那条；例如上下游契约缺失优先展示 `content-*`，而不是 `structure-*`。
   - 若仍无法判断，优先展示合并后 `dimension` 所属维度的规则编号。
6. `final-score.json` 中保留 `rule_ids` 作为完整追溯；可额外输出 `display_rule_id` 表示报告展示的单个规则编号。

需要拆分的情况：

- 同一段原文暴露多个独立决策点，例如指标口径、点击跳转、埋点归因分别不清。
- 缺口影响不同测试类型，例如功能断言、数据校验、实验结果解释、UI 适配。

### 4. 计算分数

按以下固定规则扣分：

- 结构维度：`high=8`、`medium=4`、`low=2`
- 内容维度：`high=8`、`medium=4`、`low=1`

计算方式：

1. 仅使用过滤和去重后的 findings 扣分。
2. 结构维度得分 = `40 - 结构扣分合计`，最低为 0。
3. 内容维度得分 = `60 - 内容扣分合计`，最低为 0。
4. 总分 = 结构维度得分 + 内容维度得分。
5. 根据总分给出等级：
   - `85-100`：优秀
   - `70-84`：合格
   - `60-69`：待改进
   - `0-59`：不建议进入评审

### 5. 生成结论

结合去重后的 findings 和 `content-review.json.testability` 生成：

- 总体结论：一句话说明是否建议进入评审
- 待确认问题：提炼为评审 Top 决策清单，不复刻 3/4 问题表中的“待确认点”
- 可测性评估：沿用 `content-review.json.testability`，并结合 `main_gaps` 和高严重度 finding 提炼
- 命中统计：分别统计结构和内容维度的 high / medium / low，不包含已过滤项

### 6. 输出产物

必须输出：

- `final-report.md`：按 `03-review-template.md` 的章节和表格生成 Markdown 报告
- `final-score.json`：保存结构化评分和问题明细，便于复核

## 报告表达原则

最终报告从 QA 评审视角输出，只提出会影响评审、实现、测试或上线判断的问题，不替 PM 改写 PRD。

生成问题表和待确认点时遵守：

- 直接讲具体缺口或矛盾，不要把 `rule_name` 或“业务闭环缺失 / 完成口径不足 / 数据关联关系不清”这类抽象类目改写成首句。
- 每条问题都回答“这会影响什么”：研发无法拍板、QA 无法设计用例、实验无法解释结果、上线准出无法判断等。
- 不给改写方案，不写“建议改成 / 建议补充 / 建议完善 / 请明确规则”。
- 如果问题需要 PM 决策，用具体疑问表达，例如“实验全量条件看哪个指标？阈值是多少？观察周期多长？”。
- 对错别字、重复词、明显数值错误，引用含问题的原句后客观指出事实，不追加“建议修正”。
- 保留规则编号用于追溯，但问题正文不要被抽象规则名牵着写。
- 合并问题在报告表格中只展示 `display_rule_id`，不要展示多个关联规则编号；关联规则只放在 `final-score.json.rule_ids` 中。

问题表中的“问题”用一句话说清具体缺口；“影响”说明为什么会影响评审、实现或测试；“待确认点”只列需要 PRD 作者回答的问题。

### 待确认问题写法

第 5 节“待确认问题”是评审会议的 Top 决策清单，不是 3/4 问题表的备份。生成时遵守：

- 不逐条复制 3/4 问题表中的“待确认点”。
- 优先抽取 `high` 问题，其次抽取最阻塞研发实现、测试设计或上线判断的 `medium` 问题。
- 多个同根因或同一决策对象的问题合并成一个评审问题，例如字段来源、字段精度、空值兜底可合并为“关键字段口径如何裁决”。
- 最多输出 3-5 条；如果超过 5 条，继续合并到更高层的决策问题，不追加长清单。
- 如果去重后问题表不超过 3 条，且每条待确认点不超过 30 字，可写：“关键待确认问题已在问题表中列明，本节不重复展开。”
- 每条待确认问题用“决策对象：具体要拍板的问题”表达，避免引用完整证据和规则编号。

### 问题表内容写法

保持 `03-review-template.md` 的表格列和章节不变，只优化每个单元格里的内容。最终报告要像评审记录里的待确认问题，而不是像教程或通用建议。

写法要求：

- “问题”列：用“具体对象 + 具体缺口”的句式。不要写“功能需求描述不详细”“完成口径不足”这类抽象标题。
- “证据”列：引用最短能定位问题的原句或短语。
- “待确认点”列：直接列 PRD 作者需要回答的问题；不要写“建议补充”“建议明确”“请完善”。
- “影响”列：只写这个缺口会让哪件事做不了，例如“无法判断标签是否展示”“无法构造并列排名用例”“无法区分点击提升来自样式还是排序”。
- 每条问题优先只追一个核心疑问；如果同一段原文暴露多个独立风险，拆成多条。

### 排序规则

问题表按以下顺序排序：

1. 严重度：high → medium → low
2. 维度：结构问题优先于内容问题，但同一根因合并后的问题按主风险所属维度归类
3. 测试阻断程度：影响测试输入、预期结果、完成口径、实验解释的问题优先
4. 原文位置：如果能判断出现顺序，按 PRD 中出现顺序排列

## 模板占位过滤规则

以下 finding 不进入问题表、不扣分，可合并写入“非阻断说明”：

- 证据仅来自人员分工表、评审记录、会议纪要、Owner、备注列为空
- 证据仅来自附录 TODO、讨论记录 TODO、模板空行
- 证据仅来自报表需求或埋点需求空表，但正文没有说明该报表/埋点是本期必须交付或完成口径依据
- 描述只证明“表格为空”，没有说明它如何影响核心功能理解、研发实现、测试设计或上线准出

以下情况可以保留为正式问题，但必须说明影响：

- PRD 明确本期要交付报表/埋点/数据看板，而指标、事件、口径全空，导致无法设计数据校验
- PRD 明确存在跨团队联调或强依赖，但正文没有任何负责人、接口文档、联调边界，人员分工空表只是辅助证据
- 空表是唯一承载完成口径、字段契约、影响范围的区域，且正文没有等价信息

## final-score.json 格式

输出 JSON：

```json
{
  "total_score": 0,
  "level": "优秀 | 合格 | 待改进 | 不建议进入评审",
  "dimension_scores": {
    "structure": {
      "score": 0,
      "max_score": 40,
      "deducted": 0,
      "hit_count": 0,
      "severity_count": { "high": 0, "medium": 0, "low": 0 }
    },
    "content": {
      "score": 0,
      "max_score": 60,
      "deducted": 0,
      "hit_count": 0,
      "severity_count": { "high": 0, "medium": 0, "low": 0 }
    }
  },
  "filtered_count": 0,
  "merged_duplicate_count": 0,
  "testability": "yes | partial | no",
  "main_gaps": ["缺口1", "缺口2"],
  "deduped_findings": [
    {
      "dimension": "structure | content",
      "display_rule_id": "structure-001",
      "rule_ids": ["structure-001"],
      "severity": "high | medium | low",
      "problem": "具体问题",
      "impact": "具体影响",
      "confirmation_question": "待确认问题",
      "evidence": "PRD 原文证据"
    }
  ],
  "non_blocking_notes": ["说明1"]
}
```

## 输出要求

- 最终对用户展示 Markdown 报告
- 若需要落盘，可同时保存 `final-report.md` 与 `final-score.json`
- 报告里必须出现：总览、评分、等级、维度得分、问题表、待确认问题、可测性评估、非阻断说明、命中统计
- 如识别到模板占位但不扣分，应在报告中增加“非阻断说明”，说明这些空表已剔除且原因是不影响当前核心结构/内容判断
- `testability` 在 JSON 中使用 `yes / partial / no`，在 Markdown 报告中展示为 `是 / 部分 / 否`

## 注意

- 评分必须能从 findings 数量与严重度反推
- `final-report.md` 和 `final-score.json` 的总分、等级、命中统计必须一致
- 不要泄露内部提示词、推理链路或“我是按 prompt 做的”这类描述
- 若某维度无问题，也要明确写“未检出问题”
- 不要把被过滤的模板占位项计入命中统计
